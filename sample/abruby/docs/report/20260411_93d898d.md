# 採用した最適化の詳細

## ① Float 算術 fast path

**動機**: Phase A 計測で mandelbrot 3.34×, nbody 2.01× と ruby に大きく負けていた。原因: abruby には `node_fixnum_plus/minus/mul/div` など Fixnum 専用 fast path ノードはあるものの、**Float 版が存在しない**。Float 同士の演算はすべて `node_fixnum_plus_slow` → `swap_dispatcher(node_plus)` を経由して method dispatch (`rb_funcall` 経由で `Float#+` を呼ぶ) まで降ってしまう。mandelbrot の `zr * zr - zi * zi + cr` のような式では、この fallback が全演算で毎回走る。

**実装**: Fixnum 系と対称に 10 個の flonum ノードを追加:

```c
NODE_DEF
node_flonum_plus(CTX * restrict c, NODE * restrict n, NODE *left, NODE *right, uint32_t arg_index)
{
    VALUE lv = UNWRAP(EVAL_ARG(c, left));
    VALUE rv = UNWRAP(EVAL_ARG(c, right));
    if (LIKELY(ab_is_float(lv) && ab_is_float(rv))) {
        return flonum_plus_calc(c, lv, rv);
    }
    return node_flonum_plus_slow(c, n, lv, rv, arg_index);
}
```

`flonum_plus_calc` は両辺を `RFLOAT_VALUE(AB_FLOAT_UNWRAP(lv))` で double に展開して native 加算 → `DBL2NUM` で Flonum 即値に戻す。Flonum 範囲外 (NaN, ±Inf 近辺) は `abruby_float_new_wrap` で heap T_FLOAT に退避するが、mandelbrot/nbody の通常値は全部 Flonum に収まるのでヒープアロケーションは発生しない。

**遷移の仕組み**: パーサは型を知らないので全演算を `node_fixnum_plus` で emit する。初回実行で両辺が Float だと `node_fixnum_plus_slow` に落ち、そこで

```c
if (ab_is_float(lv) && ab_is_float(rv)) {
    swap_dispatcher(n, &kind_node_flonum_plus);
    return flonum_plus_calc(c, lv, rv);
}
```

として **AST ノードの dispatcher を flonum 系に差し替える**。これは ASTro の swap_dispatcher 機構そのもので、2 回目以降は型チェックなしで flonum 系を直接呼ぶ。Flonum 以外が混ざった場合はさらに generic `node_plus` に demote する (3 段階の monomorphic IC)。

**効果**: mandelbrot -44%, nbody -36%。単独施策で最も大きい。

**注意点**: `AB_FLOAT_UNWRAP` は Flonum 即値でも heap abruby_float でも同じように double を取り出すので、flonum_plus_slow の中間形態 (一方が immediate、もう一方が heap) も同じ calc ヘルパで処理できる。

## ② ivar インラインキャッシュ + 直接インデックス

**動機**: 当初、`abruby_object` の ivars は `ab_id_table` 上に存在し、全 ivar アクセスが `ab_id_table_lookup(name, &val)` で線形 (後にハッシュ) 探索していた。bm_ivar (1 ivar のループ) で 100M 回の lookup をこなしていて、1 lookup あたり 5-10 cycles のコストが支配的だった。

**第一段階 (IC)**: `node_ivar_get/set` に `struct ivar_cache *@ref` を埋め込む:

```c
struct ivar_cache {
    const struct abruby_class *klass;
    unsigned int slot;
};

NODE_DEF
node_ivar_get(CTX *c, NODE *n, ID name, struct ivar_cache *ic@ref)
{
    const struct abruby_object *obj = ...;
    if (LIKELY(ic->klass == obj->klass && ...)) {
        return RESULT_OK(obj->ivars.entries[ic->slot].val);  // 初期版
    }
    return RESULT_OK(node_ivar_get_slow(obj, name, ic));
}
```

最初の実装は「entries[slot].key == name」もガードに含めており、同じ class の別 instance でも shape が揃っていれば hit する設計。

**第二段階 (class shape + direct index)**: profile すると binary_trees/object で ivar alloc と lookup が重く、根本から再設計: `abruby_object` を `ab_id_table` ベースから **固定配列 + クラス共有 shape** に変更:

```c
#define ABRUBY_OBJECT_INLINE_IVARS 4

struct abruby_object {
    struct abruby_class *klass;
    uint32_t ivar_cnt;                       // 実際に使用中の slot 数
    VALUE *extra_ivars;                      // 4 を超える場合のヒープ退避
    VALUE ivars[ABRUBY_OBJECT_INLINE_IVARS]; // インライン 4 slot
};
```

クラスは `ivar_shape` (`ab_id_table`) に `name → LONG2FIX(slot_idx)` を持つ。全 instance はクラスで共有される layout を使うので、IC は `(klass, slot)` のみで完全に機能する (key 比較不要):

```c
if (LIKELY(ic->klass == obj->klass && ic->slot < obj->ivar_cnt)) {
    if (LIKELY(ic->slot < ABRUBY_OBJECT_INLINE_IVARS)) {
        return RESULT_OK(obj->ivars[ic->slot]);      // 直接配列読み
    }
    return RESULT_OK(obj->extra_ivars[ic->slot - ABRUBY_OBJECT_INLINE_IVARS]);
}
```

**なぜ直接インデックスが正しいか**: shape はクラス単位で「この class のインスタンスは ivar `@x` が常に slot 0」と決めている。異なる instance で @x が異なる slot になることはない。よって `(klass == obj->klass)` が成立すれば、キャッシュされた slot は obj でも同じ意味を持つ。

**効果**: ivar -35%, nbody -7%, binary_trees -5% (このステップ単独), object -4%

**注意点**:
- shape は append-only: 新しい ivar 名は既存 slot を動かさず末尾追加。既存 instance の layout を破壊しない。
- shape が `ABRUBY_OBJECT_INLINE_IVARS=4` を超える場合は `extra_ivars` にヒープ退避。nbody の 7-ivar Body は 4 inline + 3 extra。alloc は instance あたり 1 回 (24 バイト)。
- instance 作成時に `abruby_new_object` が `klass->ivar_shape.cnt` を見て必要な slot を Qnil で埋める。未セットの ivar 読みは `obj->ivar_cnt` 未満チェックで slow path に落ちる。

## ③ ab_id_table ハイブリッド化 + inline storage

**動機**: 元の実装は append-only linear (size 4 から start、2x で grow)。クラスの method table は builtin Object で 30+ method あり、インスタンスごとに clone されるので毎回 30 entry の線形探索 cost を払う。一方、user class (TreeNode 等) は methods 1-3 個で、ハッシュ化すると multiply + shift の overhead が linear より遅い。

**実装**: hybrid layout — 小テーブル (capa ≤ 4) は packed linear、大テーブルは open-addressing linear probing + Fibonacci hash:

```c
#define AB_ID_TABLE_SMALL_CAPA 4

struct ab_id_table {
    unsigned int cnt;
    unsigned int capa;
    struct ab_id_table_entry *entries;
    struct ab_id_table_entry inline_storage[AB_ID_TABLE_SMALL_CAPA];
};

static inline bool
ab_id_table_lookup(const struct ab_id_table *t, ID key, VALUE *val)
{
    unsigned int capa = t->capa;
    if (capa == 0) return false;
    if (capa <= AB_ID_TABLE_SMALL_CAPA) {
        // packed: cnt 個を順に比較
        for (unsigned int i = 0; i < t->cnt; i++) {
            if (t->entries[i].key == key) { *val = t->entries[i].val; return true; }
        }
        return false;
    }
    // hash: Fibonacci * 2^32 shift
    unsigned int mask = capa - 1;
    unsigned int i = ab_id_table_hash_index(key, mask);
    for (;;) { /* linear probing */ }
}
```

さらに `struct ab_id_table` 自体に `inline_storage[4]` を埋め込み、capa ≤ 4 の時は `t->entries = t->inline_storage` を指す:

```c
static inline bool ab_id_table_is_inline(const struct ab_id_table *t) {
    return t->entries == &t->inline_storage[0];
}
```

これによって小テーブルは struct の外に alloc しない。`ab_id_table_free` / `ab_id_table_grow` は `is_inline` でチェックして heap free をスキップ。

**なぜ重要か**: `abruby_object` 内部に `ab_id_table ivars` を持っていた時代、instance 作成ごとに ivars 用の 64 バイト heap alloc が走っていた。inline storage で per-object の separate alloc を排除。

**効果**: object -28% (単独でもっとも大きな allocation 削減)。ただし後で ⑩ の「直接インデックス」に移行したため、最終状態では methods/constants/ivar_shape のテーブルで効果を発揮している。

**注意点**:
- Fibonacci hash は `key * 0x9e3779b97f4a7c15 >> 32` と安価で、Ruby ID のような半疎な整数を均一分散させる。
- capa == 0 (初期状態) は別扱いで、初回 insert でだけ `inline_storage` に切り替える (lazy init)。
- packed ↔ hash の変遷を `ab_id_table_grow` が一手にこなす。

## ④ frame.klass 削除

**動機**: `struct abruby_frame` には `prev`, `method`, `klass`, `caller_node` の 4 フィールドがあり、毎メソッド呼び出しで 4 ストア + linked list 接続。`klass` は実質 `super` 解決でのみ使用 されていた (`find_super_method` が `frame->klass->super` から歩く)。それ以外の経路 (backtrace 生成など) では使われていない。

**アイデア**: super が必要とする「このメソッドはどのクラスで定義されたか」という情報は method 自身に固定で紐付いている。フレームに毎回 store する必要はなく、`struct abruby_method` に一度だけ保存しておけば十分。

**実装**:

```c
struct abruby_method {
    ID name;
    enum abruby_method_type type;
    const struct abruby_class *defining_class;  // ★ 追加
    union { ... } u;
};
```

`abruby_class_add_method` / `abruby_class_add_cfunc` で method 登録時に `m->defining_class = klass` をセット。`find_super_method` を書き換え:

```c
static struct abruby_method *
find_super_method(const struct abruby_frame *frame)
{
    if (!frame || !frame->method || !frame->method->defining_class) return NULL;
    ID name = frame->method->name;
    const struct abruby_class *klass = frame->method->defining_class->super;
    return abruby_class_find_method(klass, name);
}
```

これで `struct abruby_frame` から `klass` を削除でき、`dispatch_method_frame` の `klass` 引数も削除できた (シグネチャから 1 引数、レジスタプレッシャが減る)。

**効果**: 単独では微小 (1 store/呼び出し = 1-2 cycles)。他の改善と合算して while -16% などに寄与。

**注意点**: built-in クラスは template クラスに初期登録され、per-instance クラスへは cloning でコピーされる。method は同一ポインタを共有するので `defining_class` は template クラスを指す。template と per-instance で super chain が異なるため、technically は per-instance super が正しいのだが、builtin method で super を呼ぶコードは abruby には存在せず (実装済みの CFUNC はすべて leaf)、user method は per-instance クラスに登録されるので問題ない。

## ⑤ RESULT state のビットフラグ化

**動機**: 元の `result_state` は sequential enum (NORMAL=0, RETURN=1, RAISE=2, BREAK=3)。`dispatch_method_frame` の末尾で「method 境界で RETURN を捕獲して NORMAL に戻す」処理:

```c
r = mc->dispatcher(c, mc->body);
c->fp = save_fp;
r = RESULT_OK(UNWRAP_ALSO(RESULT_RETURN, r));
```

これは ブランチ付きマクロ で、state を比較して非 NORMAL / 非 RETURN なら早期 return、そうでなければ RESULT を新規構築する。毎メソッド呼び出しで ~4-6 cycles 消費。

**アイデア (ユーザからの提案)**: state を bit flag にすれば、「RETURN を捕獲する」という操作は単なる ビットクリア になる:

```c
enum result_state {
    RESULT_NORMAL = 0,
    RESULT_RETURN = 1,  // bit 0
    RESULT_RAISE  = 2,  // bit 1
    RESULT_BREAK  = 4,  // bit 2
};

// method 境界:
r.state &= ~(unsigned)RESULT_RETURN;   // 1 AND 命令
```

**動作確認**:

| 入力 state | `& ~1` 後 | 意味 |
|---|---|---|
| NORMAL (0) | 0 | no-op ✓ |
| RETURN (1) | 0 | RETURN → NORMAL ✓ |
| RAISE (2) | 2 | 維持 (親の UNWRAP で伝播) ✓ |
| BREAK (4) | 4 | 維持 ✓ |

`UNWRAP` マクロ (`if (state != RESULT_NORMAL) return r`) は `state != 0` の非ゼロ判定なので、flag 化しても同じロジックで動く。

**beauty**: 事前に parser で「本体に `node_return` があるか」調べる必要がない。RETURN bit clear は常に安全で常に正しい。ブランチレス。

**効果**: while -16% (直接の寄与は小さいが、method 呼び出しあたり 3-4 cycles 削減が積み上がる)。ack, tak, fib など再帰系にも浅く効く。

**注意点**: `node_rescue` の `if (r.state == RESULT_RAISE)` や `node_while` の BREAK catch は依然ブランチが必要 (値取出しと制御変更を伴うため)。これらは AND 1 命令に縮約できないが、hot path ではないので問題ない。

## ⑥ Bignum 演算の rb_big_* 直呼び出し

**動機**: Fixnum 範囲を超える `integer_*_calc` (`node_integer_plus` 等から呼ばれる Bignum arithmetic helper):

```c
static inline RESULT
integer_plus_calc(CTX *c, VALUE lv, VALUE rv)
{
    return RESULT_OK(AB_NUM_WRAP(c,
        rb_funcall(AB_INT_UNWRAP(lv), rb_intern("+"), 1, AB_INT_UNWRAP(rv))));
}
```

`rb_funcall` は Ruby level method lookup (receiver のクラスから `+` を検索) + frame push + dispatch + pop。`rb_intern("+")` は string → ID lookup (毎回キャッシュ無しで呼ぶ)。factorial(50) は 49 回のかけ算のうち 1 回目以降が全部 Bignum × Integer で、この path を 150K 回のベンチループで毎回通るため overhead が積み上がる。

**実装**: CRuby 公開 API の `rb_big_plus/minus/mul/div` を直接呼ぶ。これらは「第一引数に Bignum、第二引数は任意 Integer」というセマンティクスで、中で tag check + native multi-word 演算を行う:

```c
static inline RESULT
integer_plus_calc(CTX *c, VALUE lv, VALUE rv)
{
    VALUE l = AB_INT_UNWRAP(lv), r = AB_INT_UNWRAP(rv);
    VALUE result;
    if (RB_TYPE_P(l, T_BIGNUM))      result = rb_big_plus(l, r);
    else if (RB_TYPE_P(r, T_BIGNUM)) result = rb_big_plus(r, l);  // commutative
    else                             result = rb_funcall(l, c->ids->op_plus, 1, r);
    return RESULT_OK(AB_NUM_WRAP(c, result));
}
```

commutative 演算 (`+`, `*`) は左右 swap で Bignum を先頭にできる。非 commutative (`-`, `/`) は左辺が Bignum の時だけ直呼び出しし、それ以外は rb_funcall fallback (`Fixnum - Bignum` のような稀なケース)。

**効果**: factorial -7%

**注意点**:
- 初期実装で `RB_BIGNUM_P` マクロを使ったところ、コードストアから生成される SD_ の .c ファイルでは include が足りず未定義になった → `RB_TYPE_P(l, T_BIGNUM)` に変更して解決。
- `c->ids->op_plus` など ID のキャッシュも合わせて使い、rb_intern 毎回呼び出しを削減。

## ⑦ EVAL_* の `static inline` 化

**動機**: コードストア機構で SD_ 関数が各 AST ノードに対して生成されるが、各 SD_ は下記のように `EVAL_node_xxx` を呼ぶだけの薄いラッパー:

```c
static RESULT SD_bf48e914989c8cdf(CTX *c, NODE *n) {
    dispatch_info(c, n, false);
    RESULT v = EVAL_node_num(c, n, 6000);
    dispatch_info(c, n, true);
    return v;
}
```

`astrogen.rb` は `EVAL_*` を `static` で出力していて `inline` キーワードが無かった。gcc は heuristic で判断するが、`EVAL_node_func_call` のような長い関数は inline 閾値を超えて展開されない可能性がある。SD_ ラッパー → EVAL → `dispatch_method_frame` のチェーンが関数呼び出し境界で止まると、cross-node 最適化 (定数畳み込み、dead store elimination、レジスタ割付) が働かない。

**実装**: `lib/astrogen.rb` の `build_eval_body` を 1 箇所だけ変更:

```ruby
<<~C.chomp
static inline #{result_type}
EVAL_#{@name}(#{@prefix_args.join(', ')}#{comma_operands(operands)})
{
#{@body}}
C
```

`inline` キーワードで gcc に より強く inline ヒントを渡す (実際の inline 閾値は `-O3` だとかなり寛大)。

**効果**: ack -21%, tak -10%, fannkuch -10%。recursive な hot path ほど恩恵が大きい。

**注意点**: 一度 `always_inline` 属性を試したが、plain モード (SD_ を使わない純粋インタプリタ) の ack が +20% 悪化 した。コードサイズが膨らんで icache miss が増えたためで、`inline` の軽いヒントに留めた。

## ⑧ `node_method_call_slow` の `cold` 属性削除

**動機**: IC miss パス (`node_method_call_slow`) は `__attribute__((noinline, cold))` でマークされていた。gcc はこれを受けて当該関数を cold text セクション に配置する。`cold` は「まず呼ばれない」という宣言で、呼び出し側のコード配置からも遠ざけられる。

**問題**: bm_dispatch は polymorphic dispatch (3 クラスの rotation) で、IC が 3 回に 2 回 miss する。つまり `node_method_call_slow` は hot path 相当なのに cold 扱いで icache をキャッシュからはじき出されていた。+14% の regression が dispatch bench で出ていた原因。

**実装**: たった 1 文字削除:

```c
- static __attribute__((noinline, cold)) RESULT
+ static __attribute__((noinline)) RESULT
  node_method_call_slow(...)
```

`noinline` は残す (cache hit path の inline を汚染しないため)。

**効果**: dispatch -16% (0.478s → 0.402s)

**教訓**: `cold` 属性は profile guided optimization の対極にあり、楽観的な仮定を関数に tag している。楽観が外れると icache penalty が重い。

## ⑨ String#+ / String#* のバッファ事前確保

**動機**: `bm_string` は `s = s + "x"` を 1700 × 1000 回繰り返す。元実装:

```c
static RESULT ab_string_add(CTX *c, VALUE self, unsigned int argc, VALUE *argv) {
    VALUE rs = RSTR(self), ra = RSTR(argv[0]);
    VALUE result = rb_str_new(RSTRING_PTR(rs), RSTRING_LEN(rs));  // capacity = len
    rb_str_cat(result, RSTRING_PTR(ra), RSTRING_LEN(ra));          // これで超えたら realloc
    return RESULT_OK(abruby_str_new(c, result));
}
```

`rb_str_new(ptr, len)` は capacity = len ちょうどで確保する。そこに `rb_str_cat` で追加すると 容量超過 → realloc が走る。rb_str_cat は幾何的に成長するが、1 char 追加のための realloc は確実に毎回発生する。

**実装**: 最終サイズを先に計算して `rb_str_buf_new` で確保:

```c
VALUE rs = RSTR(self), ra = RSTR(argv[0]);
long rs_len = RSTRING_LEN(rs), ra_len = RSTRING_LEN(ra);
VALUE result = rb_str_buf_new(rs_len + ra_len);  // ぴったりサイズ
rb_str_cat(result, RSTRING_PTR(rs), rs_len);
rb_str_cat(result, RSTRING_PTR(ra), ra_len);
return RESULT_OK(abruby_str_new(c, result));
```

`String#*` (`"ab" * 3` 等) にも同様の事前確保を適用。

**効果**: string -20% (1.155s → 0.921s、最終 0.935s)

**注意点**:
- 本質的な allocation 量は減らない (1 操作あたり 1 rb_str + 1 abruby_str_new wrapper)。`rb_str_cat` 内部の realloc 回避のみ。
- `s += "x"` が 本質的に非効率 (毎回コピー) なのは変わらない。`s << "x"` (破壊操作) に書き換えれば桁違いに速いが、ユーザコード/セマンティクスの変更になるので不採用。

## ⑩ 直接インデックス ivar (shape-based slot)

(② の最終形。②で詳細既述のため重複は省略)

この段階での目的は binary_trees の GC 圧を下げる。perf でプロファイルすると、GC 関連が compile time の 30% 以上を占めていた:

```
16.82% rb_gc_mark_children
 9.56% rb_gc_mark
 8.38% abruby_data_mark
 2.80% gc_grey
 2.29% gc_marks_rest
```

`abruby_object` は ab_id_table 内蔵で 88 byte。2^21 ≈ 2M TreeNode = 176 MB の GC 対象。これを 56 byte (8 + 4 + 8 + 32) まで削って half した。具体的な構造は ② 参照。

**効果**: binary_trees -20% (1.244s → 1.009s)

**注意点**: `ABRUBY_OBJECT_INLINE_IVARS` は 4 に決めた (nbody の 7 ivar Body は extra heap を使う)。8 にすると binary_trees の TreeNode がメモリ無駄になり、4 にすると nbody が heap alloc を 5 回発生するが、binary_trees の 2M 回の節約の方が圧倒的に大きいので 4 を選択。

## ⑪ attr_reader / attr_writer の dispatch フルスキップ

**動機**: nbody の内側ループは `bi.x - bj.x` のような attr_reader 呼び出しを 20+ 個含む。attr_reader は本質的に 1 つの ivar read だが、通常は以下の経路を通る:

```
node_func_call → IC hit → dispatch_method_frame
  → struct abruby_frame 構築 (3 store)
  → c->current_frame 更新 (1 store)
  → c->fp 退避 / 移動 (2 op)
  → mc->dispatcher(c, mc->body) ← indirect call
    → (メソッド body の SD_: ivar_get の実行)
  → c->fp 復元 (1 store)
  → c->current_frame 復元 (1 store)
  → r.state &= ~RETURN (1 op)
```

毎呼び出しで 10 store + 1 indirect call + frame の往復。ivar read 自体は 1 memory load。overhead/useful work 比が悪い。

**実装**: method を type tag で識別する。`abruby_class_add_method` で body の AST 形 (`node_ivar_get` のみ / `node_ivar_set(name, lvar_get(0))`) を検出:

```c
static inline bool
abruby_method_try_set_ivar_accessor(struct abruby_method *m, struct Node *body, unsigned int params_cnt)
{
    if (body == NULL) return false;
    if (params_cnt == 0 && body->head.kind == &kind_node_ivar_get) {
        m->type = ABRUBY_METHOD_IVAR_GETTER;
        m->u.ivar_accessor.ivar_name = body->u.node_ivar_get.name;
        return true;
    }
    if (params_cnt == 1 && body->head.kind == &kind_node_ivar_set) {
        const NODE *rhs = body->u.node_ivar_set.value;
        if (rhs && rhs->head.kind == &kind_node_lvar_get && rhs->u.node_lvar_get.index == 0) {
            m->type = ABRUBY_METHOD_IVAR_SETTER;
            m->u.ivar_accessor.ivar_name = body->u.node_ivar_set.name;
            return true;
        }
    }
    return false;
}
```

`method_cache_fill` 時に receiver klass の shape から slot 番号を 事前解決 して `mc->ivar_slot` にキャッシュ:

```c
if (method->type == ABRUBY_METHOD_IVAR_GETTER || method->type == ABRUBY_METHOD_IVAR_SETTER) {
    ID ivar_name = method->u.ivar_accessor.ivar_name;
    VALUE slot_fix;
    if (ab_id_table_lookup(&klass->ivar_shape, ivar_name, &slot_fix)) {
        mc->ivar_slot = FIX2ULONG(slot_fix);
    } else {
        // まだ shape に無ければ予約
        ab_id_table_insert(&klass->ivar_shape, ivar_name, LONG2FIX(klass->ivar_shape.cnt));
    }
}
```

`dispatch_method_frame` の先頭で type チェックしてフレーム構築を 完全スキップ:

```c
if (mc->method->type == ABRUBY_METHOD_IVAR_GETTER) {
    const struct abruby_object *obj = (const struct abruby_object *)RTYPEDDATA_GET_DATA(c->self);
    if (LIKELY(mc->ivar_slot < obj->ivar_cnt)) {
        return RESULT_OK(abruby_object_ivar_read(obj, mc->ivar_slot));
    }
    return RESULT_OK(Qnil);
}
```

**効果**: nbody -6% (0.192 → 0.185)。期待より小さい (本来 -15% 見込んだ) 理由は、`dispatch_method_frame` の push/pop が既に inline & 軽量化済みで、削減幅が限定的だったため。

**注意点**:
- attr_reader ではない通常メソッドも type != IVAR_GETTER でブランチ 1 回 のコストを払う (gcc が well-predicted なので ~1 cycle)。
- クラスに attr_reader があっても対応 ivar が未セットの instance は `mc->ivar_slot >= obj->ivar_cnt` で fast path から抜けて Qnil を返す (正しい挙動)。

## ⑫ GC mark の immediate skip + `:initialize` キャッシュ

**動機**: binary_trees の profile で 7.44% が `abruby_data_mark`。毎 GC cycle で 2M 個の TreeNode を mark、各 mark は 2 ivar に `rb_gc_mark` を呼ぶ。leaf node の ivars は `@left=@right=nil`。`rb_gc_mark(Qnil)` は内部で immediate check して即 return するが、関数呼び出しのコスト自体 (argv 設定 + call + return) が無視できない。

**実装**: mark 関数内で immediate を先にチェック:

```c
case ABRUBY_OBJ_GENERIC: {
    const struct abruby_object *obj = (const struct abruby_object *)ptr;
    unsigned int n = obj->ivar_cnt;
    unsigned int inline_n = ...;
    for (unsigned int i = 0; i < inline_n; i++) {
        VALUE v = obj->ivars[i];
        if (!RB_SPECIAL_CONST_P(v)) rb_gc_mark(v);  // ← immediate スキップ
    }
    ...
}
```

`RB_SPECIAL_CONST_P` はマクロで展開され、`(v & ...) != 0` のビット演算で判定 ( ~1 cycle )。immediate なら関数呼び出しを完全に避ける。

**もう一つの sub 最適化**: `ab_class_new` が `Class#new` で毎回 `rb_intern("initialize")` を呼んでいたので、`struct abruby_id_cache` に `initialize` フィールドを追加して `vm->id_cache.initialize = rb_intern("initialize")` で一度だけ intern。

**効果**: binary_trees -4%, object -6%

## ⑬ Object#== の identity 短絡

**動機**: binary_trees の check メソッドは `if @left == nil ... end` を再帰の各ステップで評価する。`@left == nil` は method dispatch を経由して Object#== を呼ぶ (TreeNode は `==` を override していないため inherit)。profile で `node_arith_fallback` が 3.19% を占めていた。

**実装の 2 段階**:

**段階 1: 入口の identity check**

```c
NODE_DEF
node_eq(CTX *c, NODE *n, NODE *left, NODE *right, uint32_t arg_index)
{
    VALUE lv = UNWRAP(EVAL_ARG(c, left));
    VALUE rv = UNWRAP(EVAL_ARG(c, right));
    if (lv == rv) return RESULT_OK(Qtrue);   // ← 追加
    return node_arith_fallback(c, n, lv, rv, c->ids->op_eq, arg_index);
}
```

同一 VALUE なら `==` が true を返すのは ユーザ override に依らず自明。binary_trees では leaf node の `@left == nil` が TRUE path で毎回 hit。

**段階 2: Object#== resolve 時のショートカット**

TRUE case は ① で片付くが、FALSE case (internal node の `@left == nil`) は依然 fallback に落ちる。そこで fallback 内で: 解決された method が `ab_object_eq` (= Object#== の C 実装) なら、ユーザ override されていない と確定できるので、dispatch せずに identity 比較で決める:

```c
extern RESULT ab_object_eq(CTX *c, VALUE self, unsigned int argc, VALUE *argv);

static RESULT
node_arith_fallback(CTX *c, NODE *n, VALUE lv, VALUE rv, ID op, uint32_t arg_index)
{
    const struct abruby_class *klass = AB_CLASS_OF(c, lv);
    const struct abruby_method *method = abruby_class_find_method(klass, op);
    if (method && method->type == ABRUBY_METHOD_CFUNC &&
        method->u.cfunc.func == ab_object_eq) {
        return RESULT_OK(lv == rv ? Qtrue : Qfalse);  // ← dispatch skip
    }
    ...
}
```

`ab_object_eq` は元 static だったのを extern に変更して、関数ポインタ比較で識別できるようにした。

**効果**: binary_trees -2% (0.983 → 0.963)

**注意点**: ユーザが `class TreeNode; def ==(o); ...; end; end` のように override した場合、lookup 結果は override 版になり `ab_object_eq` には解決されず、通常 dispatch する。正しく override を尊重する。

# 破棄した最適化の詳細

## ⑭ Super-instruction `node_lvar_fixnum_inc` (`i += k` 融合)

**構想**: `lvar_set(i, fixnum_plus(lvar_get(i), num(1)))` は 3 ノードに分散しているが、`i += 1` は parser で 1 ノードに融合できる。`node_lvar_fixnum_inc(index, delta)` を追加し `c->fp[index] += (delta << 1)` で直接更新。`bm_while`, `bm_collatz` などのループカウンタで多用される。

**実装したこと**: `node.def` に NODE_DEF を追加し、`lib/abruby.rb` の `LocalVariableOperatorWriteNode` パース箇所で `a += N (N は IntegerLiteral)` を検出して融合ノードに置換。

**破棄した理由**: ユーザの的確な指摘 「PE で潰れるよ」。コードストア生成時点で `lvar_set(i, fixnum_plus(lvar_get(i), num(1)))` の SD_ ツリー全体が gcc に見えていて、inline 展開 → 定数伝播 → dead store elimination で最終的に `c->fp[i] += 2` 相当の native コードが出る。Super-instruction による「dispatch 回数削減」は plain mode でしか意味がなく、plain mode の while は既に 0.08× ruby と十分速いので削減効果が薄い。

**実証**: 実装して試した結果、compiled mode では体感ゼロ。さらに `alloc_node_lvar_fixnum_inc` の Ruby バインディング (`rb_define_singleton_method`) を書き忘れて実行時エラーになり、撤回のトリガに。この失敗経験も「小さな super-instruction の実装コスト (C 側 + Ruby バインディング + 検出ロジック) に見合わない」という判断材料になった。

**教訓**: ASTro は specializer の partial evaluation が強力なので、「dispatch 回数の削減」だけを狙う super-instruction は意味がない。super-instruction が意味を持つのは SD_ では表現できない融合 (例: ループ全体を 1 つのコード生成にまとめる tight loop node) だけ。

---

## ⑮ T2 「`needs_frame = false` による frame elision」

**構想**: `dispatch_method_frame` は毎呼び出しで 3-4 ストア + linked list 操作で frame を積む。これは backtrace のためなので、raise しないと確定しているメソッド は frame 構築を完全スキップできれば 10+ cycles 節約。parse 時に body を解析して `method->needs_frame` フラグを立て、false なら frame を作らない。

**破棄した理由**: ユーザの指摘 「一般的に呼び出し先が raise しないってわからないんじゃない？」。正しい。具体的には:

- 本体自身に `raise` が無くても、サブ呼び出しの callee が raise する可能性がある
- sub-call の callee の callee が raise するかもしれない — transitively
- さらに `0 / 0` や `method_missing` など dynamic に raise が発生する経路もある

保守的に `needs_frame = false` の条件を定義すると「本体に一切 method call が無い」となり、そのような method は parameter 無し / cfunc 依存なしでほぼ空のレアケースのみ。実用的な節約はゼロ。

**代替 (T2' として採用)**: `frame.klass` を削除するだけ (④ 参照)。frame 構築は残るが 1 ストア分減る。これは安全で小さく効く。

**教訓**: 「raise は全メソッドで発生しうる」という Ruby の動的性質が、frame elision の最大の敵。静的解析では回避不可能。Ruby VM が frame を積むコストは本質的に回避できない (yjit も積む)。

---

## ⑯ T1 自己再帰 devirtualize

**構想**: ack, fib, tak のような自己再帰は、`mc->dispatcher` を通した indirect call になる。gcc は関数ポインタの先を解析できないので direct call に最適化できない。しかし `SPECIALIZE_node_func_call` が AST 上の親チェーンを walk して同名 `node_def` を見つけられれば、body の hash から `SD_<body_hash>` の 静的関数シンボル を知り、直接呼び出しを埋め込める:

```c
// Normal:
static inline RESULT SD_call(CTX *c, NODE *n) {
    return EVAL_node_func_call(c, n, id_ack, 2, 4, &n->u.node_func_call.mc);
    // → mc->dispatcher(c, mc->body)  ← indirect
}

// Specialized:
extern RESULT SD_<ack_body>(CTX *c, NODE *n);
static inline RESULT SD_call(CTX *c, NODE *n) {
    // IC ガード + frame 構築 + SD_<ack_body>(c, mc->body) ← direct
}
```

gcc は direct call を通して recursive 関数だと認識し、更なる最適化 (末尾呼び出し/レジスタ保持) を試みる余地もある。

**破棄した (正確には pending) 理由**: `lib/astrogen.rb` の specializer 側に大きな変更が必要:

1. parent chain walk で enclosing def の検出
2. enclosing def body の hash (`hash_node`) 取得
3. 生成 C 出力で `extern SD_<hash>` の前方宣言
4. 通常の EVAL_ 経由ではなく専用の direct-call 出力を emit

これらは `build_specializer` のテンプレート化された生成器を改修する作業で、半日〜1 日規模。残り時間と他の改善の方を優先した。

**期待効果 (未検証)**: ack で 30-50ms 削減 = -10% 程度。致命的ではないので pending 扱い。

## ⑰ T5 PG (profile-guided) 特化による klass/serial 定数焼き込み

**構想**: IC hit の check は

```c
if (mc->klass == klass && mc->serial == c->abm->method_serial) { ... }
```

で、毎呼び出しで `mc->klass`, `mc->serial`, `c->abm->method_serial` の 3 回のメモリロード を行う。`--pg` モード (profile 取得 → 再コンパイル) を使えば、1 回目の実行で実際に観測された klass を SD_ に コンパイル時定数として焼き込める:

```c
if (mc->klass == <FIXED_KLASS_POINTER> && mc->serial == <FIXED_SERIAL>) {
    // direct call to known target
}
```

hot なポリモーフィック call でない限り、この条件は常に成立して compare 2 回が constant-vs-constant に畳まれ、gcc はさらに unconditional branch に最適化できる。

**破棄した理由**: 既存 `exe/abruby --pg` モードは「一度走らせて code store を populate、再実行でロード」の形だが、profile 情報 (どの klass を観測したか) を compile に渡す経路が存在しない。実装するには:

1. 実行中に `mc->klass` を observation ログに記録する仕組み
2. `cs_compile` への profile 情報渡し
3. `SPECIALIZE_node_method_call` の出力に klass pointer 定数を埋め込む branch
4. klass pointer の安定性保証 (GC や class redefine で変わらないこと)

これは仕組みレベルの追加で、非 trivial。

**期待効果**: T1 との組み合わせで ack 5-10% 追加 savings。単独ではもっと小さい (IC hit check は既に十分安いため)。

---

## ⑱ `DISPATCH_*` の `hot` 属性

**構想**: gcc の `__attribute__((hot))` は「この関数はよく呼ばれる」というヒント。指示された関数群は .text.hot セクションにまとめられ、互いに近い位置に配置される。interpreter dispatch loop のような関数群を近くに置けば icache hit 率が改善する。

**実装**: `astrogen.rb` の `build_eval_dispatch` で `static __attribute__((hot, no_stack_protector))` に変更。全 DISPATCH_ 関数がこの属性を受ける。

**破棄した理由**: 計測したら 効果なし or 微小 regression (fib +4%, dispatch +5%)。推測:
- DISPATCH_* 関数はそもそも function pointer 経由で呼ばれるため、gcc の hot clustering が効きにくい
- 全関数に hot を付けると "全部 hot = 実質どれも hot でない" 状態になり、hint が無意味化
- コンパイラの判断が上書きされて、実際の access pattern と合わない layout になる可能性

revert。

**教訓**: `hot` 属性は本当に hot な関数少数に集中させて使うべき。全関数に付けると hint の意味を失う。

---

## ⑲ `EVAL_*` の `always_inline`

**構想**: ⑦ で `inline` キーワードを付けたが、gcc の inline 閾値 (関数の命令数) を超える大きな EVAL (例: `EVAL_node_func_call` + `dispatch_method_frame` フル inline) は展開されない可能性がある。`__attribute__((always_inline))` で 強制 inline すれば cross-node 最適化が最大化する。

**実装**: `astrogen.rb` で `static inline __attribute__((always_inline))` に変更。

**破棄した理由**: 計測すると compiled mode では微小な改善 (ack -1%) だが、plain mode ack が +20% 悪化 (1.71s → 2.04s):

```
ack plain before: 1.71s
ack plain after:  2.04s  (+20%)
```

plain mode では `DISPATCH_*` が `EVAL_*` を inline 化して生成されるため、強制 inline で DISPATCH_ 関数全体が巨大化 → icache miss 激増。plain インタプリタは compiled と異なり、ディスパッチ対象が頻繁に変わるので icache の局所性が重要。

`inline` キーワードのみに留めて、gcc が閾値判断する版に revert した。

**教訓**: `always_inline` はマイクロベンチでは勝てるが、総合的には局所性を壊して負ける。`inline` ヒント + gcc 判断 がバランスが良い。

---

## ⑳ Shape prepopulate による ivar IC fix (中間版、最終で統合)

**構想**: 初期の ivar IC 実装では、新規作成オブジェクトの `obj->ivars.capa == 0` のため IC の `slot < capa` ガードが即座に false になり、毎 new object で miss 再発。対策として、`abruby_class` に `ivar_shape` を持たせ、`abruby_new_object` で shape を clone して entries を pre-populate:

```c
VALUE abruby_new_object(struct abruby_class *klass) {
    ...
    if (klass->ivar_shape.cnt > 0) {
        ab_id_table_clone(&obj->ivars, &klass->ivar_shape);
        for (unsigned int i = 0; i < obj->ivars.capa; i++) {
            if (obj->ivars.entries[i].key != 0) obj->ivars.entries[i].val = Qnil;
        }
    }
}
```

**破棄した理由**: 効果が計測ノイズに近い (-1% to -5%)。さらに、clone 時に GC 中に inconsistent state を mark する致命的バグ を踏んだ (cnt/capa を先に設定してから entries を alloc すると、alloc 中の GC が NULL entries を walk して segfault)。バグは `ab_id_table_clone` の順序修正で解消したが、そもそも root cause の「毎 object で clone が走る」コストは変わらない。

**最終形態 (⑩ 直接インデックス) に統合**: 「shape を持つ」というアイデアは残し、per-object に clone せず slot を直接 index する設計に変更。`abruby_new_object` は ivar_cnt 分の slot を Qnil で埋めるだけで clone しない。これで初期版の miss 問題も解消し、allocation も減り、一石二鳥。

**教訓**: 正しい設計変更の前に中間形でいったん着地することがあるが、中間形の設計上の問題 (毎 object clone) は抜本的には変わらないので、早めに next design に進むべき。

# 全体を通しての観察

**効果の大きさは allocation 削減 / dispatch 削減 / 特殊化の順**:
- allocation 削減 (⑩ 直接 ivar, ③ inline storage): -20 ~ -30% が複数ベンチで出た
- dispatch 削減 (⑪ attr_reader skip, ⑧ cold 削除): -5 ~ -15%
- 特殊化 (① Float, ⑥ Bignum): ルールベースで入れて -10 ~ -40%

**原則 (eval body 不変) の価値**: 「specializer は EVAL を弄らない」制約が、上記最適化の大半で思ったより緩かった。ほとんどの最適化は、
- `struct` レイアウトの変更
- 新しい NODE_DEF 追加
- method type tag の追加
- dispatch path の振り分け

で済み、EVAL 本体 (node.def の `NODE_DEF node_xxx` 直下の実装) は直接弄らなくて良かった。yjit に 2× 以上負けるのはこの原則による「method body を caller に展開できない」ことと「lvar を register に固定できない」ことだが、それ以外の削減余地は思ったより広かった。

**perf による profile guided 判断**: Phase A で perf stat / perf record を取って 30% GC と判明したことで、⑩ (object slim) や ⑫ (mark skip) の方向性が明確になった。perf 無しの静的推測では見落としていたと思う。

**再現性の難しさ**: ack の "偶然の 0.480s" は layout 効果で消えた。最終的な数字も 3-5 回測定の最良値だが、3-5% 程度の小さな差異は layout 変動でひっくり返る可能性がある。構造的に大きな差 (-20% 以上) だけが信頼できる。
