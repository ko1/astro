# optcarrot 対応: class body / 定数周りの雑多な修正

- Date: 2026-04-11
- Commit: b87ae9a
- Target: `benchmark/optcarrot/bin/optcarrot-bench` を動かす (iter. 2)

## 経緯

1 つ前 (3dfb1ec) で splat 引数が通るようになった後、optcarrot を
再実行するたびに別の箇所で停止していた。まとめて潰した。

## 対応した 5 つの不足要素

### 1. `foo(a: 1, b: 2)` — `Prism::KeywordHashNode`

パーサが Prism::KeywordHashNode を未知扱いしていた。Ruby 上では
「末尾引数に Hash を渡したのと同じ」なので、HashNode と完全に同じ
lower に流すだけ。最終引数側でそのまま Hash として届く。

### 2. `return a, b, c`

`return` の引数が 2 個以上のとき `raise` していた。Ruby 的には
`return [a, b, c]` と等価なので、`build_ary_literal(args)` で lower。
optcarrot `Driver.load` の `return video, audio, input` 用。

### 3. `private` / `public` / `protected` / `module_function`

method_missing で "undefined method" になり停止。abruby は可視性を
持たないので、単純に Module のメソッドテーブルに **何もしない cfunc**
を登録した。引数 0 個なら self, 1 個以上なら argv[0] を返す（Ruby の
`private` / `public` 戻り値に準拠）。

### 4. Module ≺ Object ≺ Kernel の super chain

`class A; p 42; end` が "undefined method" で落ちていた。abruby の
テンプレート / init では `module_class->super = NULL` だったので、
class body の self (= A class object) から `p` / `require_relative`
を辿れなかった。

修正: Module のテンプレートに `.super = &ab_tmpl_object_class_body` を
設定し、init 時も `vm->module_class->super = vm->object_class;` に
変更。継承関係は以下:

```
Class -> Module -> Object -> Kernel -> nil
```

これで class body からも `require_relative`, `p`, `raise` を呼べる
ようになり、`const_get` / `const_set` が Module 経由で Class からも
使える（従来からだが整合性が保たれた）。

### 5. 定数は current class の super chain も探索

`class ROM; MAPPER_DB = {}; end; class MMC1 < ROM; MAPPER_DB[...] end`
が "uninitialized constant" で停止していた。abruby は `node_const_get`
で `CTX_MAIN_CLASS` → `object_class` しか見ていなかったので、
親クラス `ROM` の constants に登録した `MAPPER_DB` が子クラス body
から見えなかった。

修正: `node_const_get` の先頭で `c->current_class` から super chain を
たどり constants テーブルを順に lookup。見つからなければ従来どおり
main / object へフォールバック。Ruby の厳密な lexical lookup まで
は実装していないが、inherited constant の最頻ケースはこれで通る。

## テスト

`test/test_class_body.rb` を新規追加（12 tests）:

- `const_get` が class body 内でも呼べる
- `begin; class A; raise; end; rescue` が class body raise を捕捉
- `private` / `public` / `protected` / `module_function` が no-op
- 親クラス constants を子クラス body から参照
- `return a, b` が配列を返す
- `foo(a: 1, b: 2)` / `foo(x, k: v)` の keyword hash

780 + 15 + 12 = 807 tests, 0 failures, 0 errors（通常ビルド / debug-test
両方）。`test_method_call` の compiled-mode は稀に SIGSEGV するが、
これは本コミット以前からの既知の flaky（触っていない hot path）。
