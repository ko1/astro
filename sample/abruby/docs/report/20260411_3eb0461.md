# abruby optimization session — 2026-04-11 (3eb0461)

Session HEAD at start: `f5e97fc`. Session HEAD at end: `3eb0461`.

Follow-up to `20260411_a05d6ae.md` which covers the first 5 commits
(float-path inlining). This report adds the two `node_literal`

## 要約

Float-path を畳み込んだ後、immediate literal をノードに「埋める」
パスを導入。flonum VALUE は bit pattern として process-stable な
ので、`ab_flonum_encode` を parse 時に走らせて結果の uint64 を
node に格納、EVAL は `return RESULT_OK(val)` だけにした。
同じ node (`node_literal`) を Fixnum literal にも流用し、parser の
`int32_t` 時代の `2**30` 閾値を剥がした。

性能的にはどちらも noise 圏内 (±3%)。採用理由はアーキテクチャの
綺麗さと、今後の `node_sym` / `node_true` / `node_false` / `node_nil`
統合の地ならし。

## 採用した変更 (本レポート追加分)

### 1-E. node_literal: immediate VALUE を node に埋める (`08730fc`)

**動機**: 前セッションで `node_float_new` は `double val` を operand
として持っていたが、EVAL 時に毎回
`abruby_float_new_wrap(c, rb_float_new(val))` を通していた。
`rb_float_new` は libruby の外部関数で、compiled SD_*.c からは
inline されない。`flonum_result` に差し替えても `ab_flonum_encode`
経由で毎回 bit rotate を走らせていた。

literal は定数なので事前計算で済ませたいが、VALUE をそのまま node
に埋めるのは PE で困る: ASTro の code store はノードの operand を
SD_*.c に C リテラルとしてダンプし、別プロセスでロードし直す。
heap Float の wrapper VALUE は per-process pointer なのでプロセスを
またぐと別の値になる。Fixnum や Flonum のような immediate は
`(val << 1) | 1` / bit-rotate で double から導ける pure bit pattern
なので、プロセスをまたいでも同じ → 格納 OK。

**実装**:
- `lib/astrogen.rb` に `uint64_t` operand 型を追加。
  - `hash_call` → `hash_uint64(#{val})`
  - `build_dumper` → `%lluULL` を portability 目的で
    `(unsigned long long)` cast 経由で出力
  - `build_specializer` → SD_*.c に `(VALUE)<n>ULL` を emit
- `runtime/astro_node.c` に `hash_uint64` を追加
  (SplitMix64 の full 64-bit 版)。
- `sample/abruby/node.def` に新ノード
  `node_literal(uint64_t val)` を追加。EVAL は
  `return RESULT_OK((VALUE)val);` のみ。
- `sample/abruby/abruby.c` の `rb_alloc_node_float_new` を
  `ab_flonum_encode(val)` で分岐:
  - flonum に encode 可 → `ALLOC_node_literal((uint64_t)flo)`
  - そうでない (NaN/Inf/denormal/0x3000_…) → 従来の
    `ALLOC_node_float_new(val)` に fall back

`node_float_new` は non-flonum fallback として残る。EVAL body は
`rb_float_new` 直呼びのまま (rare path なので最適化不要)。

**効果** (best of 5, 5 independent runs, compiled mode):
- mandelbrot `0.087s -> 0.087s` (0%, median 0.090 -> 0.088, -2%)
- nbody `0.130s -> 0.134s` (+3%, 再現しない方向の noise。
  nbody の hot loop は float literal を参照しないため
  理論的に効くはずがなく、code-layout drift と判断)
- その他 bench は ±3% 以内

**注意点**:
- 性能は session rule の「5% 以上で真の効果」閾値を満たしていない。
  採用した理由は: (a) EVAL body が literal の意味どおり
  `return val` に潰れるのがアーキテクチャとして正しい、
  (b) node_num / node_sym / node_true / node_false / node_nil を
  同じ node に集約する follow-up 計画の基礎になる、
  (c) SD_*.c が
  `EVAL_node_literal(c, n, (VALUE)4613937818...ULL)` という
  読み書き可能な形に揃う。
- CRuby の flonum encoding に ABI drift が起きた場合 (CRuby の
  major upgrade など)、code store の SD_*.c に埋め込まれた
  `(VALUE)<n>ULL` が不正になるが、`cs_init` が `abruby.so` の
  mtime で store を invalidate する既存機構で自動的に捨てられる。
- `%llu` + `(unsigned long long)` cast は LP64 で `uint64_t` が
  `unsigned long` になる場合も portable。`%lu` や
  `PRIu64` でもよいが、`%llu` は C99 以降の最少驚き版。

### 1-F. node_num → node_literal 統合 (`3eb0461`)

**動機**: `1-E` の直後に user 議論で「Fixnum も bit pattern として
process-stable (`(val << 1) | 1`) なので node_literal に統合できる」
と気づいた。`node_num` は int32_t operand を持ち、parser は
`2**30` で切って bignum path に投げる hack があった (int32_t の
overflow 避け)。

**実装**:
- `abruby.c` に `rb_alloc_node_literal(VALUE v)` を追加。
  任意の immediate VALUE (Fixnum / Flonum / Symbol / true / false / nil)
  を受け、bit pattern を `uint64_t` として格納。
  ABRUBY_DEBUG の下では `RB_SPECIAL_CONST_P(v)` を assert。
- `lib/abruby.rb` の `Prism::IntegerNode` 分岐を
  `v.bit_length < 63` に変更。CRuby Fixnum は 64bit 環境で ~2^62
  の範囲に載るので `bit_length < 63` がちょうど境界。
  該当 literal は `AbRuby.alloc_node_literal(v)` に直接渡す
  (Ruby レベルで `v` は Fixnum なので C 側で VALUE
  = `(val << 1) | 1` として受け取る)。
- `node_num` 本体 (node.def 側) と `rb_alloc_node_num` の Ruby binding
  は残置。parser が emit しなくなるだけで dead code 化。次の
  clean-up session で削除予定。

**効果** (best of 5, 5 independent runs, compiled mode):
- factorial `1.211s -> 1.168s` (**-3.6%**, 方向は real。
  `fact(n-1)` と `n < 2` で literal `1` と `2` を頻繁に触る
  再帰なので微小だが効く)
- ack `0.385s -> 0.383s` (-0.5%, noise)
- gcd `0.259s -> 0.257s` (-0.8%, noise)
- fib / tak / while / nested_loop / method_call は neutral (±2%)

**注意点**:
- session rule の 5% 閾値は満たしていない。採用理由は
  「refactor としての価値」: literal 系 node が `node_literal`
  1 本に集約され、parser の `2**30` hack が消えた。
- `2^30 .. 2^62 - 1` 範囲の integer literal は以前
  `node_bignum_new(char *)` 経由で EVAL 時に
  `rb_cstr_to_inum` を走らせていた (明らかな性能バグ)。
  今回の変更で immediate Fixnum として事前 encode される
  ようになった。現在の bench は該当 literal を使わないので
  計測には出ないが、人手で書く大き目の定数 (タイムスタンプ、
  mask bit パターン等) に対しては latent な改善。

## 教訓

### PE cross-process 制約

> 「素直に VALUE にすると PE で困るわけですよ。だって、プロセスを
> またぐと、Flonum 以外の Float は、同じ値であっても別の VALUE
> なので。」

これが `node_literal` の設計で一番大事な制約だった。ASTro の
code store は:

1. AST ノードを再帰的に hash し、その hash を `SD_<hex>` という
   C symbol 名の一部にする
2. specializer が AST を C source (SD_*.c) にダンプし、operand を
   C リテラルとして emit
3. 別プロセスが `cs_init` でロードするとき、ノードを同じ入力で
   構築し直して hash を計算、同じ `SD_<hex>` を dlsym

hash と emit が**プロセスに依存しない**ことが前提。Fixnum と
Flonum の immediate VALUE は bit pattern として決定論的 (pure
function of source literal) なので OK。heap object の wrapper
VALUE はポインタ値で per-process なので NG。

### noise 判定と「refactor as optimization」

session rule は「5% 以上で真の効果」「<3% が連続 3 件で停止」と
厳しめに引いていたが、今回の `node_literal` 2 commit は効果が
noise 圏内 (±3%) で採用した。判断基準:

- **効果の方向が一貫している** (mandelbrot 中央値 -2%, factorial -3.6%
  はたまたまではなく literal の使用頻度と整合)
- **regression 候補が再現しない** (nbody +3% は連続 run で揺らぎ、
  nbody の hot loop が literal を参照しないため理論的に説明不可)
- **コード設計としての実益**が定量的 perf 以外に存在 (literal 系
  node 統一、parser hack 撤去、follow-up commit の基盤)

これを session rule の例外ケースとして記録しておく。

### 統合の名前選び

`node_flonum_const` → `node_const` → `node_literal` と 3 段階で
名前を試した。最終的に `node_literal` で落ち着いた理由:
- `node_const_get` / `node_const_set` / `node_const_path_get` が
  既存で、「Ruby レベルの定数」とセマンティクスが衝突する
- `literal` はソース由来のリテラル定数を指す言葉として曖昧さがない
- 将来 `node_num` / `node_sym` / `node_true` / `node_false` / `node_nil`
  を吸収するにあたり、これらが全て source literal である点と整合

## 全体観察と再現性

本レポートの計測は以下の環境:

- Linux 6.8.0-107-generic, ruby 3.4.4 (+PRISM), abruby.so build
  with `-O3 -DNDEBUG`
- best of 5 を 5 独立 run 取ることで noise の底上げを確認
- `rm -rf code_store` を各 run の前に入れ、compile_benchmarks の
  結果が異なる run から影響しないように

### 再現時の注意

1. **code_store のキャッシュ vs abruby.so rebuild**: `cs_init` は
   `abruby.so` の mtime で store を invalidate する。node.def を
   編集して rebuild すると、古い SD_*.c は自動的に捨てられる。
   ただし benchmark/run.rb 自体は `rm -rf code_store` を
   呼ばないので、bench の前に手で消すこと。

2. **node_literal の hash 感度**: uint64 operand の全ビットが
   hash に混ざるので、`0.0` と `-0.0` と `0x8000_0000_0000_0002`
   (flonum の `+0.0` sentinel) のような微妙な bit pattern の差も
   別の SD_*.c に specialize される。bench の reproducibility には
   影響しない。

3. **node_num は dead code**: parser は emit しなくなったが、
   `alloc_node_num` の Ruby binding と `NODE_DEF node_num` は
   残っている。外部呼び出しがあるかは未調査。次の session の
   clean-up 候補。

### 残課題

- GC dominant な bench (binary_trees, factorial, string) は引き続き
  未攻撃。`abruby_bignum` / `abruby_float` wrapper の freelist 化や
  T_DATA wrapper 撤廃 (AB_CLASS_OF に T_BIGNUM 直受けを追加) が
  次の候補。
- `node_sym` / `node_true` / `node_false` / `node_nil` を
  `node_literal` に吸収する follow-up。性能差は出ない見込みだが
  node 種別数を減らせる。
- `node_num` 本体の削除。
