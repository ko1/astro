## 0. 前提 bug fix (採用)

### abruby: fix ivar-accessor inheritance for initialize and super  (`ea6248e`)

**動機**: make test を走らせたら test_ivar_in_method と test_method_override が
segfault、test_super_initialize が assertion 失敗。原因は直前の 2 commit:

- `f8df623` が `def initialize(x); @x = x; end` 形式の initialize を
  attr_writer として IVAR_SETTER に畳み込むようにした
- `e68d947` が abruby_object の inline ivars を縮小、heap extra_ivars を導入
- しかし `ab_class_new` は `init->type != CFUNC` で `init->u.ast.body` を
  盲目的に読んでおり、IVAR_SETTER に畳まれた initialize では union
  の `ivar_accessor.ivar_name` (ID 値) を NODE* として EVAL して crash
- さらに `dispatch_method_frame` の IVAR_GETTER/SETTER 経路は
  `mc->ivar_slot` を cache fill 時の klass で解決していたが、super 経由で
  親クラスの IVAR_SETTER を呼ぶ場合 `mc->klass` (defining class) と
  `obj->klass` (receiver class) が一致せず、slot がずれて `@x` が
  TcSA のレイアウトで書かれて TcSB 側では nil に見える

**実装**:
1. `ab_class_new` で `init->type` を switch で分岐、IVAR_SETTER 時は
   `abruby_ivar_set(obj, ivar_name, argv[0])` で receiver shape に書く
2. `dispatch_method_frame` の IVAR_GETTER/SETTER fast path で
   `mc->klass != obj->klass` を検出したら receiver の shape で
   slot を再解決 (setter は登録も)。hit 時は LIKELY 分岐で既存性能を維持

**効果**: 780 tests が通る状態に復帰。性能への影響は LIKELY 分岐のみで
測定不可 (=noise)。以降の最適化の前提。

**教訓**: 型 union を扱う hot path (ab_class_new, dispatch) を追加するとき、
新しい method type を足したら **全ての union アクセス箇所を洗う** こと。
単体 smoke test (`./exe/abruby -e '...'`) では既存 class の再定義パターンだけ
通って IVAR_SETTER detection が発動しないケースがあった。test suite 全部を
最初に走らせて baseline が clean か確認する癖が効いた。

## 1. 採用した改善

### 1-A. precompute float literals (`2ac5352`)

**動機**: perf on bm_mandelbrot で `__GI_____strtod_l_internal` が 7.71%
の #1 hot symbol。`node_float_new` の EVAL body が
`abruby_float_new_wrap(c, rb_float_new(strtod(str, NULL)))` で、毎呼び出し
source 文字列を parse し直していた。mandelbrot の inner loop には
`0.0 2.0 3.0 4.0` の 4 つの float literal があり、それぞれ数千万回評価される。

**実装**:
- `lib/astrogen.rb` に operand type として `double` を追加 (hash/dump/
  specialize の 3 メソッド + `runtime/astro_node.c` の `hash_double`)
- `node.def` の `node_float_new` 引数を `const char *str` → `double val` に
- `rb_alloc_node_float_new` (abruby.c) を `NUM2DBL(num)` で受ける形に
- `lib/abruby.rb` の Prism::FloatNode 変換で `.to_s` を外す

EVAL body は `return abruby_float_new_wrap(c, rb_float_new(val));` だけ
になり、flonum-representable な値 (mandelbrot の全リテラル) は rb_float_new
も immediate 返しで allocation ゼロ。

**効果**: mandelbrot **-16.3%** (0.147 → 0.123), nbody -8.6%, hash -5.9%,
ivar -4.8%。他 noise。

**注意点**: astrogen の operand 型拡張は今回初めてだったが変更は小さく
済んだ。`hash_double` は union reinterpret で uint64 → hash_uint32 を
通す、雑だが NaN 以外で安定。

### 1-B. inline flonum encode/decode (`5cd4f12`)

**動機**: 1-A 後の perf で `rb_float_value` 8.75%, `rb_float_new` 3.15% が
top2。どちらも libruby の外部関数呼び出しで、flonum の bit-rotate で
済む処理に毎回 call/ret を払っていた。

**実装**: CRuby の `internal/numeric.h` の `rb_float_flonum_value` /
`rb_float_new_inline` を参照し、builtin/builtin.h に

- `ab_flonum_value(VALUE v)` — flonum VALUE → double (inline bit rotate)
- `ab_flonum_encode(double d)` — double → flonum VALUE or 0 (out-of-range)
- `AB_FLOAT_TO_DOUBLE(VALUE v)` — flonum fast path + heap float fallback

を `static inline` で追加。`flonum_plus/minus/mul/div_calc` を
AB_FLOAT_TO_DOUBLE に、`flonum_result` を `ab_flonum_encode` ベースに変更。
out-of-range (NaN/Inf/denormal/0x3000_) は `rb_float_new +
abruby_float_new_wrap` に fallback。

**効果**: mandelbrot **-12.2%** (0.123 → 0.108), nbody **-8.8%** (0.171 →
0.156)。累積 mandelbrot -26.5%。

**注意点**: flonum encoding は CRuby 実装依存。`USE_FLONUM` が OFF に
なるビルドでは壊れるが、abruby は flonum 前提の既存 code (`RB_FLONUM_P`
チェックが全域に散在) なのでここも同じ前提で問題なし。

### 1-C. always_inline (`5276069`)

**動機**: 1-B 後の mandelbrot perf で `AB_FLOAT_TO_DOUBLE` が 6% の独立
symbol として出現。`static inline` 宣言だけでは compiler が inline を
見送っていた (compiled SD_*.c ごとに別 translation unit のため決定が
分散)。

**実装**: `AB_FLOAT_TO_DOUBLE`, `ab_flonum_value`, `ab_flonum_encode` に
`__attribute__((always_inline))` を追加するだけ。

**効果**: mandelbrot **-7.4%** (0.108 → 0.100), nbody **-10.9%** (0.156 →
0.139)。累積 mandelbrot -32%, nbody -26%。

**注意点**: always_inline を雑に撒くと code size が膨張し I-cache ミスで
逆効果になるケースがあるが、今回対象は 5-20 行の tiny helper のみで
回帰なし。

---

### 1-D. inline flonum fast path in node_fixnum_* (`a05d6ae`)

**動機**: 1-C 後の mandelbrot perf で `node_fixnum_mul_slow` 5.28%,
`node_fixnum_plus_slow` 3.80% が top2。parser は `<num> + <num>` の型を
知らず optimistic に `node_fixnum_*` を発行するので、float の inner loop
(mandelbrot, nbody, Float + Float) は毎呼び出し cold function を呼び、
そこから `swap_dispatcher` で flonum 版に鞍替えしようとする。

しかし compiled mode では code store の SD_*.c が specialize 時の
dispatcher を**ハードコード**するので `swap_dispatcher` は no-op。結果、
compiled モードでは float hot loop が毎回 cold function を呼び続ける。

**実装**: `node_fixnum_plus/minus/mul/div/lt/le/gt/ge` の EVAL body に、
fixnum fast path の**後**に

```c
if (RB_FLONUM_P(lv) && RB_FLONUM_P(rv)) {
    return flonum_result(c, ab_flonum_value(lv) OP ab_flonum_value(rv));
}
```

を 1 ブロック追加。heap Float と integer promotion は従来の slow 関数に
残して fast path を細く保つ。fixnum 側は最上段で catch されるため
影響なし。

**効果**: mandelbrot **-10.0%** (0.100 → 0.090), nbody -2% (noise)、
nested_loop が **-36%** (0.234 → 0.150)。nested_loop は pure fixnum
workload で理屈上無関係だが、compiler の fixnum fast path 再配置で
受けた偶然のボーナスと思われる (3 回再測定しても安定)。factorial が
1.24–1.30 でばらつく (baseline 1.22) — 再現性 marginal, noise 判定。

累積 vs post-fix baseline:
- mandelbrot: 0.147 → 0.090 (-38.8%)
- nbody:      0.187 → 0.135 (-27.8%)
- nested_loop: 0.238 → 0.146 (-38.7%)

**注意点**: 今回 4 本の採用はすべて「float 演算を compiled mode でも
inline に保つ」という同じ構造問題を多面攻撃した形。compiled mode の
specializer が swap_dispatcher を観察できない制約を逆手に、fast path を
fat にして救う戦略。**compiled mode では `swap_dispatcher` が
no-op である** ことを hot 関数の最適化を考えるときは常に念頭に置くこと。

## 2. 破棄したアイデア

### 2-A. node_identity_eq specialization (構想)

**構想**: `@left == nil` が `node_fixnum_eq` slow path → `node_arith_fallback`
→ `abruby_class_find_method` (ab_id_table_lookup 9%) を毎回呼ぶ。初回の
fallback で receiver class が default Object#== を使うと判明したら
`swap_dispatcher` で `node_identity_eq` (method lookup なしの pure identity)
に切り替えて以降の lookup を回避する計画。

**試したこと**: node_identity_eq/neq を NODE_DEF で追加、node_eq_slow /
node_neq_slow で fallback 先の method が ab_object_eq なら swap して
`lv == rv` を直接返す。

**破棄理由**: plain mode では 3% 改善 (noise 範囲内) だが、compiled mode
では **0%**。原因は採用 1-D と同じ、compiled SD_* が dispatcher を
ハードコードして swap を観察しないため。ルール (5% 以上で真の効果) に
従い revert。

**教訓**: 「特定の call site を動的に specialize」パターンは compiled
mode で効かない。compiled で効かせたいなら (a) fat fast path で当該 node
本体を太らせる (1-D と同パターン)、(b) class 側の state をチェック、
のどちらかが必要。

---

### 2-B. single-slot default-eq cache in node_arith_fallback (構想)

**構想**: 2-A の compiled mode 問題を回避するため、`node_arith_fallback`
の中に single-slot 静的キャッシュを置き、`(klass, method_serial,
is_default_eq)` が match したら abruby_class_find_method をスキップする。

**試したこと**: `node.def` に `static const struct abruby_class
*arith_fb_eq_cached_klass = NULL;` 他 2 フィールドを追加、op_eq かつ
klass + serial match で identity 返却。binary_trees の
`@left == nil` を主な target にした。

**破棄理由**: perf report 上では `ab_id_table_lookup` (8.5%) が消えたが、
**binary_trees の実時間は 0.991 → 0.997** (改善ゼロ)。GC mark
(`rb_gc_mark_children` 14% + `abruby_data_mark` 8% + `rb_gc_mark` 4%) が
全体の 26% を支配しており、hot path 関数 1 つを速くしても total
に反映されなかった。ルールに従い revert。

**教訓**: perf report の symbol 占有率は「総時間に対する割合」であり、
ある関数を速くすると他の割合が膨らんで見える。**profiler の数字を
減らすのではなく、実際のベンチ時間を減らす** ことを最終判定に使う。
idea #2-B は profile の数字は動いたが時間は動かず、典型的な
"optimizing the wrong thing" だった。code store static state の
per-TU コピー問題 (SD_*.c 各々が別 translation unit で static 変数を
共有しない) も絡んでいた可能性あり。

## 3. 全体観察と再現性

### perf 環境
- Linux 6.8.0-107-generic, ruby 3.4.4 (+PRISM)
- 本セッションの beat-of-5 計測結果は `benchmark/report/20260411_*.txt`

### 再現時の注意点

1. **未コミット状態の確認**: `sample/abruby/node.c` に `static inline` を
   足す系の前セッション残骸があった。まず `git diff` を confirm する。

2. **既存の test を最初に走らせる**: HEAD が clean とは限らない。本
   セッションは最初に `make test` で crashed を見つけ、最適化前に
   bug fix が必要だった (0 節)。`make debug-test` も同様。

3. **compiled mode の swap_dispatcher 無効化**: 重要。AST node 固有の
   dispatcher を動的に切り替える pattern は plain / compile-first 両方
   で効くが、code store `--compiled-only` の SD_* が specialize 時に
   dispatcher をハードコードする構造は swap を観測しない。hot path 最適化
   を考えるときはこれを前提にする。

4. **benchmarks のノイズ**: ±3% は noise として切り捨て、5% 以上を真の
   効果と判定するルールが特に compiled mode で重要。while ベンチは
   -18% を記録したり +1% に戻ったり、run 間でブレが大きい。疑わしい
   結果は 2–3 回再測定する。

5. **code_store のキャッシュ**: benchmark を変更しなくても `node.def` を
   変えた時は `rm -rf code_store` が必要。古い .o が残ると
   `always_inline` の効果など compiler 変更が反映されない。

6. **benchmark/run.rb -n 5** を使うのは正解だった。1 回計測で
   20% 変動する bench でも、best of 5 なら多くがほぼ stable。
   ただし bignum-heavy / GC-heavy (factorial, binary_trees) は best of 5
   でも 3% 揺れる傾向がある。

### 残課題

- binary_trees / factorial / string は GC + allocation 支配 (27–30%)。
  - 現状の `abruby_bignum` / `abruby_string` wrapper は T_DATA で、CRuby
    の slot allocator に乗る。wrapper + rb_bignum の 2-allocation 構造
    を freelist / pool で圧縮できれば大きな改善が見込めるが、CRuby GC
    との協調設計が必要で本セッションの "1 commit = 1 アイデア" の粒度
    を超える。
  - 別案として abruby_header の invariant を緩めて T_BIGNUM を直接
    VALUE として扱う (AB_CLASS_OF に T_BIGNUM 分岐を追加) 戦略が考え
    られるが、全 receiver 型の class 解決を書き直す影響範囲なので
    慎重な調査が必要。
- method_call / fannkuch の DISPATCH_node_* 表示は、compiled mode で
  method body が SD_ に specialize されていない可能性を示唆。
  `--compiled-only` vs `-c` の挙動差を調べる余地がある。
- `--compiled-only` モードで任意の script を走らせると `code_store` に
  事前登録がないため null dispatcher で crash する (セッション中に遭遇)。
  これは最適化対象ではないが、debug の hazard として記録しておく。
