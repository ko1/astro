# optcarrot 対応: 多重代入ターゲット / 例外階層 / raise 拡張

- Date: 2026-04-11
- Commit: 594f5de
- Target: `benchmark/optcarrot/bin/optcarrot-bench` (iter. 3)

## 対応した不足要素

### 1. 多重代入ターゲットの拡張

optcarrot の `ppu.rb:257` は
`@vclk, @hclk_target, @cpu.next_frame_clock = ...`
の形で method setter を多重代入の左辺に置く。また `cpu.rb` では
`a[i], b[j] = x, y` 形式も散見する。

パーサの `MultiWriteNode` 処理で左辺のターゲットが LocalVariable /
InstanceVariable / GlobalVariable 以外だとエラーで停止していたので、
以下を追加:

- `Prism::IndexTargetNode` → `recv[idx...] = rhs` を
  `recv.[]=(idx..., rhs)` に lower
- `Prism::CallTargetNode` → `recv.attr = rhs` を `recv.attr=(rhs)` に
  lower（`name` は既に `=` 付きで来る）

どちらも receiver を一時スロットに退避してから右辺を評価する、
既存の call 処理の書き写し。

### 2. 例外クラスの alias

optcarrot は `class InvalidROM < StandardError` と
`raise NotImplementedError, "msg"` を使う。abruby には `RuntimeError`
しかない。

当面の解決策として、以下をすべて **RuntimeError への alias** として
Object の定数テーブルに登録:

```
Exception, StandardError, NotImplementedError, ArgumentError,
TypeError, NameError, NoMethodError, IndexError, KeyError,
RangeError, ZeroDivisionError, FloatDomainError, IOError,
LoadError, StopIteration
```

定数参照 / `class X < Y` / `rescue => e` すべてこれで parse & run する。
階層を真面目に実装するのは optcarrot が動いてから。

### 3. `raise` の 2 引数 / instance 再 raise

`raise ExceptionClass, "msg"` と `raise exception_instance` を
受け付けるように `ab_kernel_raise` を拡張:

- 0 引数: 空メッセージの RuntimeError
- 1 引数:
  - 既存 Exception instance → そのまま re-raise
  - その他（String, Integer, ...）→ 従来どおり message として扱う
    （既存テスト `raise 42` が message に 42 を渡す想定なので保持）
- 2 引数: `argv[1]` を message として扱う（`argv[0]` のクラスは無視）

### 4. method_missing エラーにメソッド名を含める

デバッグ中に「`undefined method`」しか出力されないので optcarrot の
どの method が足りないのか全く分からなかった。`method_missing_dispatch`
の exception message に `` `name'`` を埋め込むよう変更。

## テスト

`test/test_multi_assign_targets.rb` を新規追加（11 tests）:
- index target / call target / mixed local+index の多重代入
- 例外 alias の参照 / subclass / 2-arg raise / instance 再 raise

既存テストも全部通る（`make test`）。
`test_gc_pressure` の compiled-mode は flaky crash（本コミット以前からの
既知の問題、ホットパスの変更はない）。

## ベンチマーク

この変更はいずれもレアパス（parser と raise の遅い側、constants 初期化）
のみ。`fib` / `method_call` / `dispatch` は measurement 幅内で変化なし。

## 次の対応

optcarrot が次に要求する機能:

```
unsupported node: Prism::IndexOrWriteNode at ppu.rb:204
```

`(((@lut_update[nmt_bank] ||= [])[nmt_idx] ||= [nil, nil])[0] ||= []) << ...`
形式の多段 `a[i] ||= v` が ppu の LUT 構築で使われている。
`IndexOrWriteNode` とその仲間（`IndexAndWriteNode`, `CallOrWriteNode`,
`CallAndWriteNode`, 定数の `OrWrite` 版など）の support が次の課題。

潰すペースで `docs/report/YYYYMMDD_<hash>.md` に進捗を残していく方針。
