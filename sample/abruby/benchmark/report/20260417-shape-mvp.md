# shape-based ivar_cache MVP — report

Commit: 76b43ff (abruby: shape-based ivar_cache — replace klass pointer with shape_id)

## 設計

`struct ivar_cache` の `klass` pointer を 32-bit `shape_id` に差し替え。
shape_id は `(class, ivar_cnt)` のペアを識別し、`T_DATA.flags` の上位
32 bits (`flags + 4` の直 32-bit load) に格納。hot path のガードは整数
比較 1 発。

- 旧: `ic->klass == obj->klass && ic->slot < obj->ivar_cnt` (3 load, 2 cmp)
- 新: `ic->shape_id != 0 && ic->shape_id == *(uint32_t*)(flags+4)` (2 load, 2 cmp)

ivar_set は transition を想定し `post_shape_id` も cache、inline slot
の場合は fast path で shape_id の更新まで行う。

## なぜ規制が出たか

1. `ivar_set` の transition (pre_shape → post_shape) は overflow slot
   で slow path 確定。初期化時に @a..@p (16 個以上) を set すると
   **毎回 `abruby_shape_for` 呼び + ハッシュ lookup** が走る。
   optcarrot の PPU / CPU state 初期化で目立つ。

2. fast path の `ic->shape_id != 0` 追加チェックで +1 insn/call。

3. ivar_set fast path に transition case 処理が入ったので分岐コード
   が長くなった。

## 次にやるべきこと

この infrastructure を活かす**仕上げ**が必要:

1. **SPECIALIZE 焼き込み**: monomorphic な ivar_get site で
   `obj->flags+4` の literal 比較 + `obj->ivars[LITERAL_SLOT]` 直読
   を emit。9 load → 2 load に縮む最終形。

2. **Overflow slot の fast-path grow**: `extra_ivars` を inline で
   alloc する簡易分岐を追加すれば ivar_set transition が slow に
   落ちない。optcarrot 初期化の regression は消える。

3. **method_cache との統合**: `prologue_ivar_getter/setter` の guard
   も `shape_id` に寄せる。attr_reader/writer が shape-based IC を
   共有できる。

## 結論

- 正しく動く shape-based ivar_cache の土台が入った (tests 全通過)
- 最終最適化 (specialize 焼き込み) がまだ無いので optcarrot は -9%
- 次段階で 2 load 化すると pre-shape 比 +10〜15% を狙える

## 目に見える変化 (profile)

- indirect branch 数: やや増 (shape_for lookup 経路)
- 命令数: +4% (fast path の追加チェック分)
- L1 dcache miss: +5% (ic struct の alignment 変化?)

fast path の余計な insn を減らすか、specialize で literal 化するのが
次の投資対象。
