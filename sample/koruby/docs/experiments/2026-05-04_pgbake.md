# 実験レポート: ASTro PG-baked call_static (2026-05-04)

`docs/perf.md` の「今後の優先候補 1: PG-baked call_static」を実装してみたメモ。
**結論: 直接 call 化のみで +2-4 fps の安定 win。 投機的 inline (callee body
を caller TU に同梱) は size bloat (-10 fps) で逆効果。 仕切り直し中。**

スイープと AOT-PGO の本体結果は `docs/perf.md` の 2026-05-03 章に記録済。
本書はその上に乗せた "ASTro 自前 PGO" の実装メモ。

## 1. 動機

`docs/perf.md` 末尾「今後の優先候補」#1:
> **PG-baked call_static**: プロファイル後に call site の dispatcher を
> 呼出先 SD に焼き直す → YJIT 並みを狙う

加えて perf annotate (gcc-16 + AOT-PGO) で hot SD `node_func_call load_tiles 0`
(全体 19%) の中身を見ると:

```
... mc cache hit + prologue check ... ~5 cycles
... prologue_ast_simple_inl 本体 (frame setup + locals zero) ... ~25%
call *0x20(%r13)            ← mc->dispatcher 間接 call (3% on this insn)
... post-call fixup ...
```

**間接 call を直接 call に**できれば 2-3% 削れる見立て。 さらに body を
inline できれば frame setup も消える可能性あり。

## 2. 既存インフラ

実装する前に koruby のコードを精査して気づいたこと:

- `sample/koruby/prologues.h` line 134 に `prologue_ast_simple_static_inl` が
  **既に定義されているが、 どこからも呼ばれていない dead helper**。
  コメント曰く:
  > "AOT-baked variant: dispatcher is supplied as an argument. The C compiler
  > then sees a known target at the call site (when STATIC_DISP is a constant
  > function symbol), turning the indirect mc->dispatcher call into a direct
  > call which gcc can in turn inline at -O3. Used by the AOT specializer
  > when it can statically determine that a hot call site always reaches a
  > specific method body."

つまり「想定はしてあったが未配線」の状態。 wire up すれば PG-baked が動く。

- `code_store/c/SD_<hash>.c` のうち、 method 本体 (= `code_repo` entry) の
  SD は **public 非 static** (cross-TU 直接 call 可能)。 call site の SD で
  `extern VALUE SD_<body_hash>(CTX *, NODE *);` 前方宣言すれば direct call
  できる。

- `mc->body`, `mc->dispatcher`, `mc->prologue` は `--aot-compile` で program
  を走らせた直後に populate 済。 これらが `astro_cs_compile` (= SPECIALIZE)
  時点で参照可能。

## 3. 実装: koruby_gen.rb の Node#build_specializer override

`PG_CALL_NODES = %w[node_func_call node_method_call node_func_call_block
node_method_call_block]` の 4 つで build_specializer を override し、
SPECIALIZE 関数が **live `mc` を読んで判定** するように変えた。

スニペット (フル版は `koruby_gen_pgbake.rb.snippet`):

```ruby
class Node < ASTroGen::NodeDef::Node
  PG_CALL_NODES = {
    'node_func_call'         => { has_recv: false, has_block: false },
    'node_method_call'       => { has_recv: true,  has_block: false },
    'node_func_call_block'   => { has_recv: false, has_block: true  },
    'node_method_call_block' => { has_recv: true,  has_block: true  },
  }.freeze

  def build_specializer
    info = PG_CALL_NODES[@name]
    return super unless info
    build_pg_call_specializer(info)
  end
  # ... (build_pg_call_specializer は生成 SPECIALIZE_<name> の C source を返す)
end
```

生成された SPECIALIZE_node_func_call (抜粋):

```c
static void
SPECIALIZE_node_func_call(FILE *fp, NODE *n, bool is_public)
{
    const char *dispatcher_name = alloc_dispatcher_name(n);
    n->head.dispatcher_name = dispatcher_name;
    fprintf(fp, "// "); DUMP(fp, n, true); fprintf(fp, "\n");

    /* live mc を見て PG-bake 可否判定 */
    struct method_cache *mc = n->u.node_func_call.mc;
    int simple_n = -1;
    if (mc && mc->serial == korb_g_method_serial && mc->body && mc->dispatcher) {
        if      (mc->prologue == prologue_ast_simple_0) simple_n = 0;
        else if (mc->prologue == prologue_ast_simple_1) simple_n = 1;
        else if (mc->prologue == prologue_ast_simple_2) simple_n = 2;
        else if (mc->prologue == prologue_ast_simple_3) simple_n = 3;
    }

    if (simple_n >= 0) {
        /* PG-baked SD body を吐く */
        fprintf(fp, "extern VALUE %s(CTX *, NODE *);\n",
                mc->body->head.dispatcher_name);
        fprintf(fp, "static inline __attribute__((no_stack_protector)) VALUE\n");
        fprintf(fp, "%s(CTX * restrict c, NODE * restrict n)\n{\n", dispatcher_name);
        fprintf(fp, "    VALUE recv_v = c->self;\n");
        fprintf(fp, "    struct korb_proc *block = NULL;\n");
        fprintf(fp, "    struct method_cache *mc = n->u.node_func_call.mc;\n");
        fprintf(fp, "    if (LIKELY(mc->serial == korb_g_method_serial && mc->dispatcher == %s)) {\n",
                mc->body->head.dispatcher_name);
        fprintf(fp, "        return prologue_ast_simple_static_inl(c, n, recv_v, %u, %u, block, mc, %d, %s);\n",
                n->u.node_func_call.argc, n->u.node_func_call.arg_index,
                simple_n, mc->body->head.dispatcher_name);
        fprintf(fp, "    }\n");
        fprintf(fp, "    return EVAL_node_func_call(c, n, n->u.node_func_call.name, %u, %u, mc);\n",
                n->u.node_func_call.argc, n->u.node_func_call.arg_index);
        fprintf(fp, "}\n\n");
    } else {
        /* 元の codepath */
        ...
    }
}
```

optcarrot で 117/727 SDs (16%) が PG-baked になった。

## 4. 結果

RUNS=10 FRAMES=300, taskset -c 0, optcarrot 600 frames, checksum 検証 OK。

| Config (gcc-15/16, AOT -O2 -march=native)      | best   | median  | size  |
|-----------------------------------------------|-------:|--------:|------:|
| baseline (no PG)                              | 106.32 | 105.61  | 3.5MB |
| AOT-PGO 単独 (gcc-16)                         | 112.42 | 111.56  | 2.9MB |
| **ASTro PG-bake 単独 (gcc-16)**               | 107.76 | 106.80  | 4.0MB |
| **PG-bake + AOT-PGO (gcc-16)** ★              | 115.12 | **114.04** | 3.3MB |

効果分解 (gcc-16, vs no-PG baseline):

| 改善                  | gain      | size 変化 |
|-----------------------|----------:|----------:|
| AOT-PGO 単独          | +7.3 fps  | -19%      |
| ASTro PG-bake 単独    | +2.5 fps  | +9%       |
| 両方 stack            | **+9.7 fps** | -10% |

stack 率 ~95%。 **両者は異なる箇所を最適化しているのでほぼ独立に効く**。
- AOT-PGO: cold path 削除、 命令並べ替え (icache friendly)
- PG-bake: call dispatch の間接 call → 直接 call

## 5. 失敗した試行: 投機的 inline (callee body 同梱)

「直接 call の次は body inline では?」と試した版:

```ruby
if (simple_n >= 0) {
    SPECIALIZE(fp, mc->body);  # ← 追加: body の SD subtree を caller TU に同梱
    if (mc->body->head.flags.no_inline) goto pg_bake_skip;  # 再帰メソッドは skip
    ...
}
```

これで `static_disp(c, mc->body)` 呼出に対し body の static inline 定義が
同 TU 内にあるので gcc が cross-TU inline 不要で inline できる、 はず。

結果:

| Config                             | size  | median fps |
|------------------------------------|------:|-----------:|
| ASTro PG-bake (direct call)        | 3.9MB | 109.59     |
| ASTro PG-bake (body inline)        | **7.9MB** | 97.37   |
| body inline + AOT-PGO trim         | 5.3MB | 104.40     |

**全部 baseline 以下に regression**。 size bloat (3.9MB → 7.9MB) が
icache 圧で +inline 利得を全部喰い潰した。 `-Os` の -20 fps 大破壊と
同じ機序。

`docs/perf.md` で確認済の知見:
> "Aggressive force-inlining everywhere blows up register pressure and
> icache (measured 4-7× regressions on pi / sha256 / fannkuch), so we
> ship gcc's heuristic by default and opt the specific nodes that need
> it via @always_inline."

callee body inline は「特定の小さい body だけ」しか許されない、 という
ことを再確認した。 size 制限つき (例: subtree size < 50 nodes) なら
有効かもしれないが、 今回は実装せず。

## 6. 教訓

1. **既にある dead helper を見逃さない**: `prologue_ast_simple_static_inl`
   は実装当時の意図 (PG-bake) を待っていた。 wire up だけで動く。

2. **間接 call の elimination は安全 win**: +2-4 fps、 size +9%。
   icache の影響は最小。

3. **body inline は罠**: size bloat で全部消える。 LTO 無し環境では
   特に厳しい。 やるなら subtree size 制限が必須。

4. **SPECIALIZE-time に live runtime state を読める**のが ASTro 設計の
   強み。 `--aot-compile` が「program を走らせてから SD 生成」する流れ
   なので、 mc などの cache 状態を C 生成側が直接参照できる。

5. **stack は ~95%**: gcc PGO と ASTro PGO は異なる layer を最適化
   しているので両者ほぼ独立。 production では両方使うのが最適。

6. **load の影響大**: load 5 のとき +4 fps、 load 7-8 のとき +2 fps と
   noise 域に入る。 安定測定には load < 5 + RUNS=10 + FRAMES=300 が要る。

## 7. 仕切り直し方針 (将来の実装)

今回 koruby_gen.rb override は work tree でのみ実験し、 commit せずに
revert した。 本実装するなら:

a. **node.def に @canonical 機構を導入** (abruby 流): `node_call0_ast`,
   `node_call1_ast`, ... を canonical=node_func_call な variant として
   定義し、 mc 観測したら parse-time に swap_dispatcher。 SPECIALIZE は
   variant ごとに別 body を emit。 framework の対称性が高い。

b. **2-step Hash (Horg / Hopt)**: koruby は今 HORG == HOPT (= hash_node)。
   PGC を入れるなら abruby と同様に分離し、 PGSD_<Hopt>.c と SD_<Horg>.c
   を共存させる。 既存 hopt_index.txt 機構が使える (`runtime/astro_code_store.c`
   の hopt_index_load_file 等)。

c. **size limit つき inline-bake**: subtree size < 50 nodes の body のみ
   caller TU に同梱、 それ以外は direct call。 ノード数カウントのヘルパ
   (例: `node_subtree_size(NODE *)`) が必要。 abruby に類似実装があれば
   流用可。

d. **AOT-PGO とのハイブリッド**: gcc PGO の profile data を ASTro PGC
   側でも見て decision に使う (e.g., gcc PGO で hot 判定された SD だけ
   inline-bake する)。
